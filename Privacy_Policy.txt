2.8. "Hash" — the output of a cryptographic hash function (MD5, SHA-1, SHA-256, etc.) applied to a file, representing a unique fixed-length numeric identifier. A hash function is one-directional: it is technically impossible to reconstruct the original file content from its hash.

2.12. "ML Model" (Machine Learning model / neural network) — a mathematical model trained on a dataset to identify patterns and classify threats. Operates locally within the Application, without transmitting user data.

2.13. "Zero-Day Threat" — malicious software or a vulnerability for which no known signature or official patch exists at the time of discovery.

SECTION 5. TECHNICAL METHODS OF ANALYSIS AND SECURITY

5.1. General Architecture

The Armaga Application implements a multi-layered threat detection architecture combining classical and modern analysis methods. All methods described below are implemented with a priority on local processing: user data does not leave the device at any stage of analysis.

The composition of algorithms used, the number of code models, and their operational logic may change as the Application evolves. The introduction of new scanning methods does not necessarily require an update to this Policy, provided it does not result in an expansion of the personal data collected.

5.2. Layer 1: Signature-Based Analysis

5.2.1. How it works. The Application engine computes file hashes from the device's file system and compares them against a local database of known threat hashes. A hash match indicates the presence of a known malicious object.

5.2.2. Database updates. Signature databases are downloaded from the Developer's secured repository on GitHub via HTTPS and updated automatically when an internet connection is available (if the Application is running in 24/7 mode) or manually upon User request. The Developer strongly recommends regularly updating signature databases and installing Application updates as soon as they become available in the app store.

5.2.3. Scope of application. Signature analysis is applied to APK files in accessible device directories, files in external storage, and components of installed applications — subject to the relevant permissions being granted.

5.3. Layer 2: Heuristic Analysis

5.3.1. How it works. The heuristic analyzer assesses the potential threat level of applications based on algorithms designed by the Developer, primarily through analysis of the quantity and types of permissions requested by an application and their correspondence to the application's declared functionality.

5.3.2. Indicators of potential danger may include:
   — Requests for an excessive number of privileged permissions not justified by the application's functionality;
   — Permission combinations characteristic of spyware or adware;
   — Other behavioral patterns as determined by the Application's algorithms.

5.3.3. Based on the analysis results, the Application informs the User about potential risks, leaving the final decision entirely to the User.

5.4. Layer 3: ML Analysis (Neural Network)

5.4.1. The Application incorporates a local ML model (neural network) designed to detect threats not covered by signature analysis: Zero-Day malware, Spyware, and Adware.

5.4.2. Model training is carried out by the Developer independently using publicly available data. The trained model is integrated directly into the Application. All inference is performed locally on the User's device — no data for ML analysis is transmitted to external systems.

5.4.3. The composition, quantity, and architecture of ML models may be changed by the Developer without prior notice, provided such changes do not result in the transmission of new categories of personal data.

5.5. Layer 4: Real-Time Background Monitoring ("Silent Mode")

5.5.1. The Application supports a 24/7 background operation mode for continuous monitoring of device activity, aimed at detecting Adware and other malicious behavior in real time.

5.5.2. This mode is designed to minimize impact on device performance and battery consumption. The Application's policy is to avoid burdening the device or degrading its operability.

5.5.3. In 24/7 mode, the Application independently checks for signature database updates. If this mode is not activated, the Developer recommends checking for updates manually.

5.5.4. The User may disable background monitoring at any time through the Application settings. Disabling it does not affect the availability of manual scanning features.

SECTION 6. ANDROID PERMISSIONS AND THEIR PURPOSE

6.1. General Provisions

To implement the declared antivirus protection features, the Application may request a number of Android operating system permissions. Each permission is used exclusively for the purpose of ensuring the security of the User's device.

Important: the set of permissions requested may change across different versions of the Application depending on the addition, modification, or removal of features. The current list of permissions is always reflected in the Android interface upon installation or update, as well as on the Application's page in the Google Play Store. The Developer does not request permissions that are not functionally necessary.

The User may deny any permission. Denial preserves the Application's general operability but limits the depth and scope of available protection features.

6.2. Current List of Permissions and Justification

6.2.1. MANAGE_EXTERNAL_STORAGE
   (Manage all files in storage)

   Google Play classification: high-risk permission.

   Justification: This permission is standard and necessary for any antivirus application. Without it, the Application cannot scan the device's external storage (SD card, "Downloads", "Documents" folders), creating blind spots for malicious APK files.

   Specific purposes:
   — Searching external storage for APK files to scan;
   — Computing file hashes and comparing them against the signature database.

   What the Application does NOT do with this permission:
   — Does not read, copy, display, or transmit the content of user files (photos, documents, videos, audio, etc.);
   — Does not transmit file names or paths outside the device.

6.2.2. QUERY_ALL_PACKAGES
   (Query information about all installed packages)

   Justification: On Android 11 and above, without this permission the Application can only see a limited list of applications, making a comprehensive security audit impossible.

   Specific purposes:
   — Compiling a complete list of installed and pre-installed applications for display to the User and for analysis;
   — Retrieving metadata (version, permissions) for heuristic and ML analysis.

6.2.3. BIND_ACCESSIBILITY_SERVICE
   (Accessibility Service — Accessibility API)

   Google Play classification: high-risk permission.

   Justification: Phishing overlays and aggressive advertising windows are implemented through Android's system overlay mechanisms. Real-time detection and blocking requires access to the Accessibility API. This permission is activated exclusively with the User's explicit consent and may be disabled in the Application settings at any time.

   Specific purposes:
   — Detecting unauthorized overlays on top of the screen (phishing, fraudulent interfaces);
   — Identifying and blocking aggressive advertising pop-ups.

   What the Application STRICTLY does NOT do with this permission:
   — Does not read text messages, SMS, or messenger correspondence;
   — Does not intercept passwords, PINs, or payment card data;
   — Does not record screen content;
   — Does not collect any personal User data.

   Compliance declaration: The use of the Accessibility API in the Application fully complies with Google Play policy requirements — the permission is used exclusively for the purpose of device security and does not serve surveillance purposes.

6.2.4. SYSTEM_ALERT_WINDOW
   (Display over other applications)

   Justification: When an active threat is detected, the Application must immediately notify the User regardless of which application is currently open.

   Specific purposes:
   — Displaying critical security warnings over other windows.

6.2.5. PACKAGE_USAGE_STATS
   (App usage statistics)

   Justification: Aggressive adware frequently activates at the moment of app launch or switching. Correlating advertising events with specific applications requires access to usage statistics.

   Specific purposes:
   — Identifying the source of aggressive advertising overlays;
   — Generating a report on potentially dangerous applications for the User.

6.2.6. REQUEST_DELETE_PACKAGES
   (Initiate package deletion)

   Justification: After detecting a malicious application, the User must be provided with a convenient mechanism to remove it directly from the Armaga interface.

   Specific purposes:
   — Launching the standard Android system dialog for deletion of the selected application.

   Limitation: The Application cannot delete packages automatically without the User's explicit confirmation through the standard Android operating system dialog.

6.2.7. Other Permissions

   Depending on the Application version, other technically necessary permissions may be requested (for example, permissions for background service operation, internet access, etc.). The complete and current list of permissions is always available on the Application's page in the Google Play Store and in the Android device system settings. All requested permissions are exclusively dictated by functional necessity to protect the User's device.

SECTION 7. ARTIFICIAL INTELLIGENCE INTEGRATION

7.1. General Description

An AI Assistant is integrated into the Application — a security consultant for the device with a number of built-in Application control commands. As of the effective date of this Policy, the language model in use is ChatGPT (developed by OpenAI, LLC, USA) via the official OpenAI API. The Developer reserves the right to change the language model or its version at any time, provided such a change does not result in an expansion of the personal data transmitted.

7.2. How It Works and Architecture

7.2.1. The AI Assistant functions as an external consultant. The User's text query is transmitted to the AI provider's servers (OpenAI or another provider active at the time of the request) via a secure HTTPS channel. The response is returned to the User in the Application interface.

7.2.2. The AI Assistant operates on its own developer-provider's servers (OpenAI or another provider). The Application Developer does not operate its own servers for processing AI queries.

7.2.3. The processing of queries on the AI provider's servers is governed by the relevant provider's privacy policy. For OpenAI, the current policy is available at: https://openai.com/privacy.

7.3. AI Information Isolation Principle

7.3.1. The AI Assistant is completely isolated from the User's personal data. This is due both to the Developer's architectural decision and to the fundamental limitations of the Android platform on transmitting personal information to third-party services.

7.3.2. The AI Assistant never receives:
   — The content of files on the device;
   — File names or paths;
   — Personal correspondence, photos, or contacts;
   — Passwords, PINs, or other credentials;
   — Any other personal User data.

7.3.3. When executing commands (e.g., "run a scan", "update databases"), the AI receives from the Application exclusively an aggregated technical response (for example: "no threats detected", "databases updated"), but never specific file names, their content, or personal User information.

7.4. Autonomy and Control

7.4.1. The AI Assistant has no direct access to the device's system APIs. It cannot independently initiate scans, delete files, or change device settings.

7.4.2. All actions proposed by the AI Assistant are executed through predefined internal Application commands exclusively at the User's explicit request and confirmation. The AI serves as a consultant and assistant, not as an autonomous device management agent.

7.5. AI Assistant Availability

7.5.1. Access to the AI Assistant is available exclusively to authenticated Users. Authentication is currently carried out through a Google account; additional authentication methods may be added in the future.

7.5.2. An authenticated User is provided with a limited number of AI Assistant queries free of charge. The limit is increased or removed with an active subscription.

7.6. Data and OpenAI

7.6.1. The Developer does not use User query texts to train or fine-tune any AI models.

7.6.2. OpenAI's data usage policy for API interactions is available at: https://openai.com/policies/api-data-usage-policies.

SECTION 10. DATA STORAGE, PROTECTION AND SECURITY

10.1. Data Storage Location

10.1.1. All Category B data (analysis results, scan history, application metadata) is stored exclusively in the Application's secured internal storage (Sandbox) on the User's device. This data is inaccessible to other applications through Android OS mechanisms (unless the device has been rooted) and is completely inaccessible to the Developer.

10.1.2. Category A data (email) is stored exclusively within Google's infrastructure (Firebase Authentication, Google Cloud Platform). The Developer does not operate independent personal data storage servers.
